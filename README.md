# SciOly Robot Tour — Code Compliance Checker

A small web app that helps Science Olympiad Event Supervisors judge whether a team's Robot Tour code complies with the rules. The Event Supervisor uploads the official rules text and the team's source file, and the app returns a plain-English compliance report generated by Claude.

## How it works

- **Frontend:** single static page (`public/`) with two file inputs and an access code field.
- **Backend:** Express server (`server.js`) that validates the access code, rate-limits per IP, and calls the Anthropic API (`claude-sonnet-4-20250514`) via `lib/analyze.js`.
- **Logging:** every analysis appends one JSON line to `logs/usage.log` (timestamp, IP, file sizes, verdict, duration). On Vercel, logs go to stdout instead.

## Local setup

Requires Node.js 18+.

```bash
npm install
cp .env.example .env
# Edit .env: set ANTHROPIC_API_KEY and pick an ACCESS_CODE
npm start
# Open http://localhost:3000
```

## Configuration

| Variable            | Required | Purpose                                                              |
|---------------------|----------|----------------------------------------------------------------------|
| `ANTHROPIC_API_KEY` | yes      | Org's Anthropic API key. The app pays via this key.                  |
| `ACCESS_CODE`       | yes      | Shared password ESes enter in the UI before any API call is made.    |
| `PORT`              | no       | Defaults to 3000.                                                    |

## Safety model

- The Anthropic key is held only on the server — never sent to the browser.
- Requests are blocked unless the user enters the correct `ACCESS_CODE`. Distribute the code to ESes via official SciOly channels.
- The access-code check runs **before** the rate limiter so wrong codes don't consume a user's quota.
- Per-IP rate limit: **1 request per 30 seconds**. There is intentionally no daily or total cap — the tool must not stop mid-competition.
- Per-file upload cap: 500KB (Robot Tour code is tiny; this just blocks accidental huge uploads).
- Every successful or failed analysis writes a log line so the org can audit usage.

## Deploying

### Railway (recommended)

1. Create a new Railway project from this repo.
2. Set `ANTHROPIC_API_KEY` and `ACCESS_CODE` as service variables.
3. Railway auto-detects Node and runs `npm start`. Done.

### Vercel

1. `vercel.json` is already configured to route all traffic to `server.js`.
2. Set `ANTHROPIC_API_KEY` and `ACCESS_CODE` in the Vercel project's Environment Variables.
3. Deploy. Note: Vercel's filesystem is read-only, so usage logs go to function logs (visible in the Vercel dashboard) instead of `logs/usage.log`.

## Updating the system prompt

The compliance criteria live in `SYSTEM_PROMPT` inside `lib/analyze.js`. Edit there to refine what counts as a violation. The three baked-in concerns are:

1. Event parameters (target time, gate positions) passed in as library inputs.
2. Libraries that abstract navigation or motion sequencing.
3. Generic stock/template code rather than student work.

The Markdown output format (verdict line + per-concern sections) is also defined there. The first line of the response must start with `**VERDICT: PASS`, `**VERDICT: FAIL`, or `**VERDICT: NEEDS REVIEW` — the server parses this for the log.
