# SciOly Code Review Assistant

A small web app that helps Science Olympiad Event Supervisors review a team's code submission against event rules. The ES picks an event (or uploads a custom rules file), uploads the team's code folder, and the app returns a list of **questions to review with the team**, generated by Claude.

**This tool does not render verdicts.** It surfaces patterns in the code that might implicate a rule and suggests questions the ES can ask. The Event Supervisor is always the judge. This design is deliberate: Event Supervisors are typically not programmers, and a confident false "FAIL" from an AI could unfairly disqualify a compliant team.

## How it works

- **Frontend:** single static page (`public/`) with a rules-source selector, access code field, and folder picker. Each flagged question renders as a card with a priority label, what the AI noticed, why it might matter, and a suggested question for the team.
- **Backend:** Express server (`server.js`) that validates the access code, rate-limits per IP, bundles the submitted files, and calls the Anthropic API (`claude-sonnet-4-20250514`) via `lib/analyze.js`.
- **Output format:** Claude returns structured JSON (`{questions: [...]}`), parsed in `lib/analyze.js` via assistant-prefill. The frontend renders each question with static, auditable labels.
- **Logging:** every analysis appends one JSON line to `logs/usage.log` (timestamp, IP, rules source, file sizes, `questionCount`, duration). On Vercel, logs go to stdout instead.

## Local setup

Requires Node.js 18+.

```bash
npm install
cp .env.example .env
# Edit .env: set ANTHROPIC_API_KEY and pick an ACCESS_CODE
npm start
# Open http://localhost:3000
```

Test samples are included in `samples/` — pick one of the three folders there when the UI asks for a code folder.

## Configuration

| Variable            | Required | Purpose                                                              |
|---------------------|----------|----------------------------------------------------------------------|
| `ANTHROPIC_API_KEY` | yes      | Org's Anthropic API key. The app bills API usage to this key.        |
| `ACCESS_CODE`       | yes      | Shared password ESes enter in the UI before any API call is made.    |
| `PORT`              | no       | Defaults to 3000.                                                    |

## Safety & abuse controls

- The Anthropic key is held only on the server — **never** sent to the browser.
- Requests are rejected unless the user enters the correct `ACCESS_CODE`. Distribute the code to ESes via official SciOly channels.
- The access-code check runs **before** the rate limiter so wrong codes don't consume a user's quota.
- Per-IP rate limit: **1 request per 30 seconds**. There is intentionally no daily or total cap — the tool must not stop mid-competition.
- Upload limits:
  - 200 KB per individual file
  - 500 KB total across all code files in a submission
  - Up to 200 files per submission
  - Binaries, build artifacts, `.git/`, `node_modules/`, `__pycache__/`, `.vscode/`, etc. are filtered out server-side (see `lib/bundle.js`).
- Every successful or failed analysis writes a JSON log line for later audit.

## Cost (approximate)

At Claude Sonnet 4 pricing (Apr 2026: $3 / $15 per M tokens input/output):
- **~$0.02 per analysis** for typical ~50 KB code submissions
- **~$0.20 per analysis** for the maximum 500 KB bundle
- 15,000 analyses/year ≈ **$300–$3,000/year** depending on average bundle size

Biggest cost driver is code size, not the prompt or output. Consider switching to Haiku 4.5 in `lib/analyze.js` if volume grows — roughly 3–5× cheaper, with likely some loss on the subtler stock-code checks.

## Adding a new event preset

1. Drop a new `.txt` file into `rules/` (e.g. `rules/helicopter.txt`).
2. Add one line to `PRESET_DEFS` in `lib/rules-presets.js`:
   ```js
   { id: 'helicopter', name: 'Helicopter', file: 'helicopter.txt' }
   ```
3. Restart the server. The new event appears as a radio option in the UI automatically.

The UI fetches presets from `GET /api/rules`, so no frontend change is required.

## Deploying

### Railway (recommended)

1. Create a new Railway project from this repo.
2. Set `ANTHROPIC_API_KEY` and `ACCESS_CODE` as service variables.
3. Railway auto-detects Node and runs `npm start`. Done.

### Vercel

1. `vercel.json` is already configured to route all traffic to `server.js`.
2. Set `ANTHROPIC_API_KEY` and `ACCESS_CODE` in the Vercel project's Environment Variables.
3. Deploy. Note: Vercel's filesystem is read-only, so usage logs go to function logs (visible in the Vercel dashboard) instead of `logs/usage.log`.

## Tuning the prompt or output

The review criteria live in `SYSTEM_PROMPT` inside `lib/analyze.js`. It instructs Claude to:

- **Never** use verdict language (the words PASS, FAIL, VIOLATION, COMPLIANT, etc. are explicitly forbidden).
- Look at three topic areas plus anything the supplied rules text calls out:
  1. Event-specific parameters (target time, gates, target distance) being passed as arguments to a library.
  2. Libraries that appear to plan navigation, motion sequencing, or scored outcomes.
  3. Generic stock/template code that doesn't appear to be team-authored.
- Raise a question whenever a pattern plausibly implicates a rule, labeling each as **"likely"** (probably worth discussing) or **"brief"** (borderline / quick check).
- Return a single JSON object matching the schema defined in the prompt. The frontend renders this with static labels — there is no markdown parsing.

If you want to change what counts as a question, edit `SYSTEM_PROMPT`. If you want to change how questions are displayed, edit `public/app.js` (`renderQuestionCard`) and `public/style.css`. The static "What this tool did NOT check" disclaimer is in `public/index.html` and can be edited without changing any code.
